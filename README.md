# RZD Route Viewer

Веб-приложение для поиска поездов между станциями РЖД и просмотра маршрута выбранного поезда в виде вертикального таймлайна.

Проект состоит из двух частей:

- `backend` — API-обёртка над публичными HTTP-эндпоинтами сайта РЖД;
- `frontend` — клиентский интерфейс на чистом HTML/CSS/JavaScript.

## Что умеет проект

- искать станции по введённой строке;
- получать список поездов между двумя станциями;
- заранее проверять, доступен ли маршрут для каждого поезда;
- показывать только те поезда, для которых маршрут действительно найден;
- открывать детальный маршрут поезда в полноэкранном режиме;
- подсвечивать текущую станцию на таймлайне, если она приходит от API РЖД.

## Логика работы проекта

### 1. Поиск станции

Когда пользователь начинает вводить название станции:

1. frontend отслеживает ввод в полях отправления и прибытия;
2. вызов оборачивается в `debounce` с задержкой 250 мс, чтобы не отправлять слишком много запросов;
3. frontend вызывает backend-метод `GET /stations?q=...`;
4. backend обращается к эндпоинту подсказок РЖД `https://www.rzd.ru/suggests/rstation/`;
5. из ответа выбираются только основные поля станции:
   - `id`
   - `compositeRegion`
   - `searchName`
   - `suWeight`
   - `dtWeight`
6. frontend сортирует станции по релевантности и показывает выпадающий список.

### 2. Выбор станций

Важно, что приложение ожидает именно выбор из выпадающего списка, а не просто текст в поле.

После клика по станции frontend сохраняет весь объект станции в:

- `selectedFrom`
- `selectedTo`

Это нужно потому, что для дальнейших запросов используются именно `id` станций, а не их названия.

### 3. Поиск поездов

После нажатия на кнопку `Найти поезда` frontend:

1. проверяет, что обе станции выбраны из подсказок;
2. автоматически берёт **текущую дату** на стороне клиента;
3. вызывает backend-эндпоинт:
   - `GET /trains?date=DD.MM.YYYY&station_departure_id=...&station_arrival_id=...`
4. backend валидирует формат даты `DD.MM.YYYY`;
5. backend отправляет POST-запрос на эндпоинт расписания РЖД `https://www.rzd.ru/tt/train/schedule`;
6. из ответа backend возвращает нормализованный список поездов.

Возвращаемые поля поезда:

- `trainNumberRus`
- `trainNumberLatin`
- `trainStartStation`
- `trainEndStation`
- `departureStation`
- `arrivalStation`
- `passStartStationDifferTrainStartStation`

### 4. Предзагрузка маршрутов

После получения списка поездов frontend не показывает их сразу.

Сначала выполняется дополнительная проверка:

1. для каждого поезда вычисляется дата запроса маршрута;
2. дата берётся из `arrivalStation.localDateTime`;
3. формат преобразуется из `DD.MM.YYYY` в `YYYY-MM-DD`;
4. затем вызывается `GET /train-route?train_number_latin=...&date=YYYY-MM-DD`;
5. если маршрут успешно найден, он кэшируется в `routeCache`;
6. если маршрут не найден или API вернул ошибку, такой поезд скрывается из итогового списка.

### 5. Почему показываются не все поезда

Это важная часть логики проекта:

**на экран попадают только те поезда, для которых backend смог получить маршрут движения.**

То есть список результатов — это не просто все найденные поезда между двумя станциями, а только поезда с доступной детализацией маршрута.

Это сделано для того, чтобы пользователь не открывал карточку поезда и не получал пустой экран или ошибку.

### 6. Открытие маршрута поезда

Когда пользователь нажимает на карточку поезда:

1. открывается полноэкранный overlay;
2. frontend пытается взять маршрут из `routeCache`;
3. если маршрута в кэше нет, делает повторный запрос через backend;
4. полученный маршрут отображается в виде вертикального таймлайна.

Для каждой станции маршрута показываются:

- название станции;
- время прибытия;
- время отправления.

Если в данных РЖД есть признак `current_station`, интерфейс визуально разделяет:

- уже пройденные станции;
- текущую станцию;
- следующие станции.

## Как backend работает с РЖД

Backend не хранит собственные данные и не использует базу данных.

Его задача:

- принимать запросы от frontend;
- проксировать их в РЖД;
- валидировать часть входных параметров;
- нормализовать ответы;
- скрывать лишние детали внешнего API от frontend.

### Класс `RZDClient`

Внутри `backend/main.py` реализован класс `RZDClient`, который использует `requests.Session()` и общий набор HTTP-заголовков.

Это даёт:

- повторное использование HTTP-сессии;
- единый `User-Agent`;
- единое поведение для всех запросов к РЖД.

### Реализованные backend-методы

#### `GET /stations`
Ищет станции по строке `q`.

#### `GET /trains`
Возвращает список поездов между двумя станциями на указанную дату.

#### `GET /train-route`
Возвращает маршрут поезда по его `trainNumberLatin` и дате в формате `YYYY-MM-DD`.

Если маршрут не найден, backend возвращает `404`.

## Как frontend устроен внутри

Frontend написан без фреймворков.

### Основные части

- `index.html`
  - содержит разметку интерфейса;
  - содержит стили;
  - содержит основную логику работы страницы в `script type="module"`.
- `rzdApiClient.js`
  - изолирует HTTP-запросы к backend;
  - формирует URL и проверяет HTTP-ошибки.
- `server.js`
  - поднимает простой HTTP-сервер на Node.js;
  - раздаёт статические файлы;
  - реализует hot reload через Server-Sent Events и `chokidar`.

### Особенности frontend-логики

- backend URL по умолчанию вычисляется как:
  - `http(s)://<hostname>:8000`
- результаты до первого поиска скрыты;
- дата поиска не выбирается вручную — используется текущий день;
- найденные маршруты кэшируются в `Map`, чтобы не запрашивать их повторно;
- карточка поезда кликабельна целиком;
- окно маршрута можно закрыть кнопкой назад или клавишей `Escape`.

## API проекта

### `GET /stations`

Параметры:

- `q` — строка поиска станции.

Пример:

```/dev/null/stations-example.txt#L1-1
/stations?q=москва
```

### `GET /trains`

Параметры:

- `date` — дата в формате `DD.MM.YYYY`;
- `station_departure_id` — ID станции отправления;
- `station_arrival_id` — ID станции прибытия.

Пример:

```/dev/null/trains-example.txt#L1-1
/trains?date=05.05.2026&station_departure_id=2000006&station_arrival_id=2004001
```

### `GET /train-route`

Параметры:

- `train_number_latin` — идентификатор поезда из поля `trainNumberLatin`;
- `date` — дата в формате `YYYY-MM-DD`.

Пример:

```/dev/null/train-route-example.txt#L1-1
/train-route?train_number_latin=717MJ&date=2026-05-06
```

## Запуск проекта

### Вариант 1. Через Docker Compose

Из корня проекта:

```/dev/null/docker-compose-run.txt#L1-1
docker compose up --build
```

После запуска:

- frontend: `http://localhost`
- backend: `http://localhost:8000`

### Вариант 2. Локально без Docker

#### Backend

Из директории `backend`:

```/dev/null/backend-install.txt#L1-2
pip install -r requirements.txt
uvicorn main:app --host 0.0.0.0 --port 8000
```

#### Frontend

Из директории `frontend`:

```/dev/null/frontend-install.txt#L1-2
npm install
node server.js
```

После этого frontend будет доступен на порту `80`, а backend — на `8000`.

## Ограничения и особенности

- проект зависит от доступности внешних HTTP-эндпоинтов РЖД;
- структура ответов РЖД не контролируется проектом и может измениться;
- поиск поездов выполняется только на текущую дату в интерфейсе;
- база данных отсутствует — все данные запрашиваются в реальном времени;
- backend открыт по CORS для любых origin (`allow_origins=["*"]`).

## Зачем такой подход

Этот проект удобен как лёгкая витрина над данными РЖД:

- frontend остаётся очень простым;
- backend скрывает нестабильные детали внешнего API;
- логика фильтрации поездов по наличию маршрута повышает качество UX;
- кэширование маршрутов уменьшает число повторных запросов.
